# ProjectNautic 结构恢复与占领系统 — 功能规格说明书（Feature Spec）

> 版本：v1.0
> 日期：2026-07-16
> 状态：待开发（Ready for Development Agent）

---

## 一、核心愿景（Vision）

ProjectNautic 的冒险结构具有**自愈特性**：当玩家离开结构后，结构会随时间缓慢恢复到原始状态。但玩家可以通过某种行为**永久占领结构**，使其停止自愈，成为玩家的领地/基地。

这创造了一个动态张力：
- **无人探索时**：结构保持"野生"状态，被破坏后会自然恢复（适合重复挑战）
- **被占领后**：结构成为玩家领地，玩家可以改造它、利用它（适合基地建设）

## 三、模块一：结构注册（Structure Registration）

### 3.1 结构定义

每个需要自愈功能的冒险结构必须预先注册：

```yaml
# 示例：config/projectnautic/structures/sunken_temple.yml
structure_id: "sunken_temple_01"
display_name: "沉没神庙"
world: "minecraft:overworld"
# 结构边界（AABB）
bounds:
  min: { x: 100, y: 40, z: 200 }
  max: { x: 150, y: 80, z: 250 }
# 结构模板（原始状态的 .nbt 文件路径）
template: "projectnautic:structures/sunken_temple.nbt"
# 结构锚点（模板放置的基准坐标）
anchor: { x: 100, y: 40, z: 200 }
# 结构类型（影响恢复策略）
type: "dungeon"  # dungeon | outpost | ruins | temple
```

### 3.2 注册方式

支持两种注册方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **配置文件** | 手动编辑 YAML/JSON 注册 | 开发者预设结构 |
| **命令注册** | 管理员用命令框选区域并保存模板 | 动态添加新结构 |

### 3.3 模板要求

- 每个结构必须有一个对应的 `.nbt` 模板文件
- 模板中应包含结构的**完整原始状态**，包括：方块、方块实体（刷怪笼、箱子、命令方块等）、实体（如果有）
- 模板可通过原版结构方块保存，或通过 WorldEdit 导出

## 四、模块二：状态检测（Presence Detection）

### 4.1 "无人存在"的定义

一个结构处于"无人状态"，当且仅当：

```
条件 1：结构边界内没有任何玩家实体（Player）
条件 2：结构边界附近 N 格内没有任何玩家（可配置，默认 32 格）
条件 3：该结构当前未被任何玩家/队伍"占领"
```

### 4.2 检测频率

- 每 tick 检测过于昂贵，建议每 **20 ticks（1秒）** 或 **100 ticks（5秒）** 检测一次
- 检测逻辑应轻量化：只遍历结构区域内的玩家列表，而非扫描整个区域

### 4.3 状态转换图

```
                    玩家进入区域
    ┌──────────────────────────────────────┐
    │                                      ▼
┌─────────┐      玩家全部离开      ┌───────────┐
│ 野生状态  │ ◄─────────────────── │  有人状态  │
│(可恢复)  │                        │ (暂停恢复) │
└────┬────┘                        └───────────┘
     │
     │ 占领行为（见模块四）
     ▼
┌───────────┐
│ 已占领状态  │
│(永久停止恢复)│
└───────────┘
```

## 五、模块三：恢复引擎（Restore Engine）

### 5.1 触发时机

结构从"有人状态"转为"无人状态"后，**不立即恢复**，而是进入一段"冷静期"：

```
玩家全部离开 ──→ [冷静期 5分钟] ──→ 开始恢复 ──→ [恢复间隔 10秒] ──→ 恢复下一批
```

| 阶段 | 时长 | 说明 |
|------|------|------|
| 冷静期（Cooldown） | 可配置，默认 300 秒 | 防止玩家在边缘反复进出导致频繁暂停/恢复 |
| 恢复间隔（Tick） | 可配置，默认 10 秒 | 每 N 秒执行一次恢复批次 |

### 5.2 恢复内容

每次恢复应修复以下内容（按优先级）：

| 优先级 | 恢复内容 | 说明 |
|--------|---------|------|
| P0 | **刷怪笼（Spawner）** | 核心游戏机制，必须恢复 |
| P0 | **关键方块（结构框架）** | 墙壁、地板、天花板等 |
| P1 | **战利品箱（Loot Chests）** | 重新填充 loot table |
| P1 | **红石机关/门/陷阱** | 恢复原始状态 |
| P2 | **装饰性方块** | 雕像、植被、火把等 |
| P3 | **流体（水/岩浆）** | 恢复原始流体布局 |

### 5.3 恢复方式

**不是一次性恢复全部，而是每批次恢复一部分**：

```
每恢复间隔（10秒）：
  1. 对比当前世界状态 vs 模板状态
  2. 找出"缺失/被改变的方块"（diff）
  3. 从 diff 列表中取出前 N 个（如 20 个）
  4. 用 setBlock 恢复为模板状态
  5. 如 diff 列表为空，恢复完成
```

### 5.4 恢复效果（可选）

为增强沉浸感，恢复时可以附带：
- 粒子效果（如 `happy_villager`、`witch`）
- 音效（如 `block.stone.place`）
- 如果结构是水下/魔法主题，可使用主题化粒子

### 5.5 需要跳过的情况

以下情况不应恢复：
- 结构已被**占领**（见模块四）
- 方块当前状态是玩家**合理改造**（如玩家建造的工作台、放置的床）
- 方块是玩家**放置的火把**（合理的探索行为）

## 六、模块四：占领系统（Claim System）

### 6.1 占领方式

玩家通过特定行为"占领"结构，使其永久停止自愈。提议的占领方式（可选其一或多种）：

| 占领方式 | 说明 | 适合场景 |
|---------|------|---------|
| **击败Boss** | 击败结构内的Boss后自动占领 | Boss房/神庙 |
| **点亮祭坛** | 在结构中心放置特定物品/激活机关 | 神殿/祭坛 |
| **放置旗帜** | 在结构内放置旗帜/标记 | 前哨站/哨塔 |
| **命令占领** | 管理员/OP 手动将结构标记为已占领 | 调试/特殊剧情 |
| **满足条件** | 完成特定任务/成就后自动占领 | 剧情驱动 |

### 6.2 占领后的效果

结构被占领后：
- ✅ **永久停止**自动恢复（即使无人也不再自愈）
- ✅ 该结构成为**玩家领地**，玩家可以任意改造
- ✅ 可配置额外效果（如：刷怪笼停止刷怪、敌对生物不再生成、村民/友好NPC生成）
- ✅ 占领状态**持久化保存**（即使服务器重启也保留）

### 6.3 占领标识

需要一种视觉或游戏内提示，告知玩家该结构已被占领：

```
方案 A：聊天消息 — "沉没神庙已被 [玩家名] 占领"
方案 B：标题/副标题 — 进入区域时显示 "已占领领地"
方案 C：粒子边界 — 区域边界显示占领者颜色的粒子
方案 D：地图标记 — 在大地图上显示领地标记
方案 E：告示牌 — 在结构入口自动生成"已被占领"告示牌
```

### 6.4 占领解除（可选）

是否允许解除占领？

```
选项 1：永久占领（不可解除）— 最简单
选项 2：占领者可手动放弃
选项 3：其他玩家可通过某种方式"夺取"（PvP 领地战）
选项 4：占领后 N 天不活跃自动解除
```

**建议**：初期采用**选项 1（永久占领）**，最简单且最清晰。

## 七、持久化与数据存储

### 7.1 需要保存的数据

```yaml
# 每个结构的运行时状态
structure_states:
  sunken_temple_01:
    claimed: true                    # 是否被占领
    claimed_by: "player_uuid"         # 占领者 UUID
    claimed_at: 1690123456            # 占领时间戳
    last_player_exit: 1690123000     # 最后玩家离开时间
    restore_in_progress: false        # 是否正在恢复
    restore_queue: []                 # 待恢复方块队列（可选，用于断点续传）
    
  coral_ruins_02:
    claimed: false
    # ... 未占领结构只保存基本状态
```

### 7.2 存储方式

| 实现路径 | 存储方式 |
|---------|---------|
| **KubeJS** | 使用 `server.persistentData` 或自定义 JSON 文件 |
| **Fabric 模组** | 使用 `LevelSavedData` 或 `Attachment` API |
| **数据包** | 记分板/存储（能力有限，不推荐）|

## 八、配置项汇总

```yaml
# config/projectnautic/structure_restore.yml
structure_restore:
  # 全局开关
  enabled: true
  
  # 玩家检测
  detection:
    radius: 32              # 玩家检测半径（格）
    check_interval: 20      # 检测频率（ticks）
    
  # 恢复行为
  restore:
    cooldown_seconds: 300   # 玩家离开后冷静期（秒）
    interval_seconds: 10    # 恢复间隔（秒）
    blocks_per_batch: 20    # 每批次恢复方块数
    max_restore_time: 600   # 最大恢复时间（秒），超时强制完成
    
  # 占领系统
  claim:
    enabled: true
    method: "boss_defeat"   # boss_defeat | altar | banner | command
    permanent: true         # 占领是否永久
    
  # 视觉效果
  effects:
    particles: true
    particle_type: "happy_villager"
    sound: true
    sound_type: "block.stone.place"
    
  # 日志
  logging:
    debug: false
    notify_players: true    # 是否在恢复/占领时通知玩家
```

## 九、边界情况与处理

| 场景 | 预期行为 |
|------|---------|
| 玩家离开结构后马上返回 | 冷静期重置，不触发恢复 |
| 恢复过程中玩家进入 | 立即暂停恢复，保留当前恢复进度 |
| 结构被严重破坏（如TNT） | 恢复队列变长，恢复时间延长，但不会无限增长 |
| 服务器重启时正在恢复 | 恢复进度持久化，重启后从断点继续（如实现复杂可先不处理）|
| 多人同时在结构边缘进出 | 检测逻辑应稳定，不因短暂无人而误触发恢复 |
| 结构被占领后玩家继续破坏 | 不再恢复，玩家可以任意改造 |

## 十、与 ProjectNautic 整合包架构的适配

### 10.1 推荐技术路径

基于 ProjectNautic 是**整合包**（Fabric/Forge 模组包），推荐：

```
首选方案：KubeJS + 数据包混合
├── KubeJS 负责：事件监听、状态管理、玩家检测、占领逻辑
├── 数据包负责：结构模板（.nbt）、loot table、配置
└── 持久化：server.persistentData + 自定义 JSON

备选方案：独立 Fabric 模组
├── 完全自定义实现
├── 最高性能、最灵活
└── 开发成本高
```

### 10.2 文件结构建议

```
ProjectNautic/
├── config/
│   └── projectnautic/
│       ├── structure_restore.yml      # 主配置
│       └── structures/                # 结构定义
│           ├── sunken_temple.yml
│           └── coral_ruins.yml
├── kubejs/
│   ├── server_scripts/
│   │   └── structure_restore/         # KubeJS 逻辑
│   │       ├── main.js                # 主入口
│   │       ├── detector.js            # 玩家检测
│   │       ├── restorer.js            # 恢复引擎
│   │       └── claim.js               # 占领系统
│   └── startup_scripts/
│       └── structure_registry.js      # 结构注册
└── data/
    └── projectnautic/
        └── structures/                # .nbt 模板
            ├── sunken_temple.nbt
            └── coral_ruins.nbt
```

## 十一、验收标准（Definition of Done）

开发 agent 完成实现后，应满足以下验收条件：

- [ ] 已注册的结构在玩家离开后进入冷静期，冷静期结束后开始缓慢恢复
- [ ] 恢复过程是渐进的（非瞬间），每批次恢复固定数量的方块
- [ ] 恢复内容包括刷怪笼、战利品箱、结构框架等关键元素
- [ ] 玩家进入结构后，恢复立即暂停
- [ ] 玩家可以通过配置的方式"占领"结构，占领后结构永久停止恢复
- [ ] 占领状态在服务器重启后仍然保留
- [ ] 恢复过程对服务器 TPS 影响可接受（恢复时 TPS 下降不超过 5%）
- [ ] 配置化：结构定义、检测半径、恢复速度、占领方式等均可配置
