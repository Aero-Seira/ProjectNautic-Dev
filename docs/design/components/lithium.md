## 设计作用

Lithium 是 ProjectNautic 整合包的通用性能优化模组，作用于服务端和客户端双方。它通过对游戏逻辑算法（实体 AI、方块刻、红石、数据管理、物理等）的系统性重写和优化，在保持原版行为不变的前提下降低 tick 负载。Lithium 支撑整合包核心设计支柱——**性能优先**，确保服务端在多人环境下保持稳定 TPS，同时让单人客户端获得更流畅的 tick 响应。

与 Sodium（专注渲染）形成互补：Sodium 负责"画得快"，Lithium 负责"算得快"。两者共同构成整合包的基础性能层。

## 玩家可见行为

- **可发现性**：Lithium 是无缝优化的典型代表，玩家通常不会注意到它的存在，但会在以下场景间接受益：
  - 大型动物农场/村民交易所的帧率更稳定
  - 红石机械的响应更一致
  - 多人在线时服务端延迟更低
- **交互**：无直接交互界面。模组不添加 GUI、不修改设置菜单。
- **进度/奖励**：无直接进度关联。间接奖励为更稳定的游戏体验和更高的实体/红石负载容忍度。
- **摩擦**：无额外摩擦。模组开箱即用，无需玩家手动配置。
- **失败状态**：若 Lithium 的某项优化与特定模组行为冲突，可能需要手动在 `config/lithium.properties` 中禁用对应 mixin。目前尚未发现此类冲突。

## 配置意图

Lithium 的配置文件 `config/lithium.properties` 当前采用接近默认的配置策略，体现"默认即最优"的设计哲学：

- `mixin.ai=false` 等配置项保持默认状态，意味着大部分优化 mixin 均处于启用状态。Lithium 的设计原则是尽可能覆盖更多优化路径，仅在确认存在兼容性问题时才选择性禁用特定 mixin。
- 配置文件当前以默认配置为主，表明该模组在当前模组批次中未遇到需要手动调整的冲突场景。这种"零配置"方式降低了维护负担，也减少了因错误配置导致性能损失或行为异常的风险。
- 若未来添加大量自定义实体 AI 的模组（如自定义村民职业、复杂动物行为模组），可能需要重新审视 `mixin.ai` 相关设置。目前整合包定位为"基础性能与 QoL 模组"，无需此类干预。

## 兼容性与性能

- **客户端/服务端范围**：BOTH。这是 Lithium 的关键特性——同一份模组文件可用于客户端和服务端，确保单人游戏和多人服务器均获得优化。
- **存档影响**：无。Lithium 不修改存档格式，移除 Lithium 后存档完全可继续游戏。
- **已知不兼容**：当前批次中无已知不兼容模组。Lithium 以高度兼容性著称，其大部分优化通过算法替换实现而非行为修改。
- **资源成本**：Lithium 优化通常降低 CPU 和内存占用，无额外资源开销。
- **缓解措施**：配置文件 `lithium.properties` 提供了细粒度的 mixin 开关，若未来发现特定优化与某模组冲突，可针对性禁用对应 mixin 而不影响其他优化。

## 风险与开放问题

1. **Mixin 冲突排查复杂度**：若未来出现难以定位的性能或行为异常，Lithium 的大量 mixin 可能增加排查难度。建议在问题发生时首先通过 `lithium.properties` 进行二分法排查。
2. **服务端-客户端版本同步**：由于 Lithium 是 BOTH 范围模组，需确保服务端和客户端版本严格一致。整合包更新时需同步推送两端。
3. **与 KubeJS 脚本的潜在交互**：KubeJS 可能通过脚本修改实体行为或方块逻辑，若未来脚本与 Lithium 优化路径冲突，可能需要调整脚本或禁用对应 mixin。
