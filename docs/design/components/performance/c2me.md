## 设计作用

C2ME（Concurrent Chunk Management Engine）是 ProjectNautic 整合包的服务端与客户端区块性能优化核心。它通过并行化区块加载、生成、序列化与 I/O 流程，降低高视距、高密度世界中的 MSPT 与加载卡顿。该模组直接支撑整合包**性能优先**与**长线运营优先**两大设计支柱：前者确保玩家在高移动速度或多人场景下仍能获得流畅体验；后者通过优化区块系统减少长期运营中因区块膨胀导致的性能衰减。

由于 C2ME 标记为 `BOTH`，服务端与客户端需同时部署，以充分利用协议扩展（如扩展渲染距离协议）。

## 玩家可见行为

- **可发现性**：玩家通常不会直接看到 C2ME 的 UI，但会在以下场景感受到效果：
  - 快速飞行、鞘翅滑翔或乘船移动时区块加载更顺畅
  - 大规模世界生成（如预生成、探险）时服务器 TPS 更稳定
  - 高视距设置下帧率与加载延迟改善
- **交互**：无直接交互界面。配置由服务端统一控制。
- **进度/奖励**：无直接进度关联。间接奖励为更稳定的探索体验。
- **摩擦**：无额外摩擦。默认配置已面向现代硬件优化。
- **失败状态**：alpha 版本存在潜在不稳定性；与特定区块管理模组（如 `dynview`、`betterchunkloading`）共存可能导致严重故障。

## 配置意图

`config/c2me.toml` 当前全部使用 `"default"` 值，未进行覆盖。这表明整合包选择接受 C2ME 的官方默认优化策略，避免在 alpha 版本中引入额外变量。记录主要默认行为如下：

- **全局执行器并行度**：由表达式自动根据 CPU 核心数与内存计算，避免手动配置导致的过线程或欠线程。
- **区块系统**：启用异步序列化、允许 POI 卸载、同步玩家 ticket、流体后处理转为计划刻，均旨在降低内存泄漏与 MSPT 峰值。
- **世界生成优化**：启用密度函数编译、含水层优化、结构权重采样优化，加速世界生成而不改变结果。
- **无 tick 视距**：默认最大并发区块加载为 24，扩展渲染距离协议已启用。
- **I/O 系统**：使用优化实现并设置 NBT 缓存软硬限制，减少磁盘 I/O 阻塞。

所有选项保持默认，待 alpha 版本稳定后再根据实际性能数据决定是否微调。

## 兼容性与性能

- **客户端/服务端范围**：BOTH。服务端必须安装，客户端推荐安装以启用协议扩展。
- **存档影响**：理论上不改变存档格式或世界生成结果；alpha 版本仍建议在升级前备份存档。
- **已知不兼容**：`dynview`、`betterchunkloading`（discouraged）。
- **资源成本**：通过并行化换取响应性，CPU 与内存占用可能略有增加；在现代服务器硬件上收益远大于开销。
- **缓解措施**：保持默认配置；在服务器更新 C2ME 时进行完整备份；监控控制台是否有异步加载错误。

## 风险与开放问题

1. **Alpha 版本稳定性**：`0.4.0-alpha.0.115` 属于早期版本，可能存在未知的并发崩溃或区块损坏风险。需密切关注上游更新。
2. **与服务端同源原则**：C2ME 要求服务端与客户端版本严格一致，否则协议扩展可能失效。需在版本更新流程中强制同步。
3. **与 DynView / BetterChunkLoading 的误装风险**：玩家在手动添加模组时可能误装这些冲突模组，需在整合包说明中明确标注。
4. **长期存档兼容性**：C2ME 优化了 I/O 与 POI 卸载，长期运行后的存档在移除 C2ME 后是否完全兼容尚待验证。
