## 设计作用

FastSuite 将 JSON 配方匹配工作从主线程迁移到并行线程池，显著降低自动合成、漏斗、投掷器等频繁查询配方时的 tick 压力。它与 Lithium（逻辑算法优化）共同构成服务端与客户端的“算得快”层，支撑整合包“性能优先”的支柱，同时保持原版配方结果不变。

## 玩家可见行为

- **可发现性**：玩家通常不会直接感知，但在大型自动合成阵列、漏斗网络或频繁打开工作台时，会感到操作响应更及时、帧率或 TPS 更稳定。
- **交互**：无直接交互界面，无需玩家手动配置。
- **进度/奖励**：无直接关联。间接奖励是更流畅的合成与物品处理体验。
- **摩擦/失败**：若某个配方并非线程安全，FastSuite 会在达到最大查找时间后中止并报错；关闭验证的“不安全模式”可能引发崩溃或错误匹配，当前配置保持关闭。

## 配置意图

`config/fastsuite.cfg` 采用保守的默认并行策略：

- `S:"Single Threaded Recipe Types" < >`（空列表）— 当前未强制任何配方类型回退到主线程，保留最大并行收益。
- `I:"Max Recipe Lookup Time"=25` — 单个配方查找最大 25 秒后中止并记录错误，防止异常配方无限阻塞。
- `B:"Lock Crafting Input Stacks"=false` — 不在输入栈上启用调试锁，避免额外开销。
- `B:"Unsafe Mode"=false` — 关闭“未经验证全部并行”模式，仅对通过安全性验证的配方启用并行，降低崩溃风险。

## 兼容性与性能

- **客户端/服务端范围**：BOTH
- **存档影响**：无。不修改存档格式或配方数据
- **已知不兼容**：当前批次中无已知不兼容模组
- **资源成本**：使用额外工作线程分担主线程 CPU 负载；内存开销主要来自并行任务栈，通常较小
- **缓解措施**：保持 `Unsafe Mode: false`；当遇到线程安全相关崩溃时，可将对应 `recipe type` 加入 `Single Threaded Recipe Types` 进行回退

## 风险与开放问题

1. **自定义配方的线程安全性未知**：部分模组或数据包添加的配方可能在并行匹配下行为异常，需要通过测试覆盖常见配方类型。
2. **与动态配方脚本的交互未验证**：若 KubeJS/LootJS 在运行时热修改配方，FastSuite 的缓存与并行路径是否保持一致需回归检查。
