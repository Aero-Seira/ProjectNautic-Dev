## 设计作用

LootJS 是 ProjectNautic 整合包 KubeJS 魔改框架的扩展插件，专门用于通过 KubeJS 脚本修改 Minecraft 战利品表（Loot Tables）。它支撑整合包**魔改即基础设施**的设计支柱：整合包的资源平衡、掉落调整、阶段化解锁、自定义奖励箱等设计很大程度上需要对战利品表进行程序化控制，LootJS 提供了比直接编辑 JSON 数据包更灵活、可版本控制的脚本化方案。

该模组属于 `integration` 分类：它本身不添加独立游戏内容，而是扩展 KubeJS 与 Minecraft 战利品系统之间的集成能力。

## 玩家可见行为

- **可发现性**：普通玩家不会直接感知 LootJS 的存在；其效果体现在 KubeJS 脚本对战利品表的修改结果中（如怪物掉落、宝箱内容、钓鱼收获、方块破坏掉落等）。
- **交互**：无直接交互界面。效果通过脚本间接影响玩家可获得的物品。
- **进度/奖励**： LootJS 是实施阶段化奖励与平衡调整的核心工具。例如：
  - 调整特定生物的掉落以配合进度解锁
  - 修改地牢宝箱内容以控制稀有资源获取节奏
  - 为自定义事件添加战利品奖励
- **摩擦**：无额外摩擦。脚本错误可能导致掉落异常，但这是脚本层面的问题而非模组本身。
- **失败状态**：脚本语法错误或战利品表 ID 不存在时，`server_scripts` 会在控制台报错，相关修改不生效。

## 兼容性与性能

- **客户端/服务端范围**：BOTH。战利品表修改主要在服务端生效，但客户端需同步安装以保持一致。
- **存档影响**：战利品表修改对已有世界即时生效（新击杀、新开箱应用新规则），已存在的掉落物品不受影响。修改脚本时需在更新说明中明确标注。
- **已知不兼容**：当前批次中无已知冲突。与直接修改战利品表的数据包可能产生叠加或覆盖效果，需谨慎管理。
- **资源成本**：脚本在战利品事件触发时执行，复杂条件判断可能轻微增加事件处理耗时。建议避免在每次掉落事件中执行重量级计算。
- **缓解措施**：建立脚本性能规范，避免全局监听所有战利品事件；优先使用具体过滤器（如生物类型、战利品表 ID）。

## 风险与开放问题

1. **脚本复杂度风险**：LootJS 非常灵活，若无规范约束，战利品脚本可能迅速变得难以调试。需建立"每个战利品修改须对应设计文档"的流程。
2. **与数据包的优先级**：未来若同时使用数据包和 LootJS 修改战利品表，需明确优先级规则以避免预期外行为。
3. **阶段化掉落设计**：LootJS 是实现"阶段式解锁"的重要工具，但当前阶段系统尚未落地，需在后续设计中明确其使用边界。
