## 设计作用

spark 是 ProjectNautic 整合包的性能剖析与诊断工具，支持对客户端、服务端与代理进行 CPU 采样、内存堆转储、GC 监控与网络流量分析。它是整合包**长线运营优先**与**性能优先**设计支柱的关键运维组件：在开发阶段帮助定位启动与运行瓶颈，在运营阶段帮助管理员诊断 TPS 下降、内存泄漏与网络异常。

spark 标记为 `BOTH`，服务端与客户端均需部署，以确保在任意侧发生性能问题时都能快速取证。

## 玩家可见行为

- **可发现性**：普通玩家通常不会直接使用 spark；管理员与开发者可通过命令生成性能报告。
- **交互**：
  - `/spark profiler` 启动/停止 CPU 采样并生成可分享的链接报告
  - `/spark heap` 进行内存堆转储
  - `/spark gc` 查看垃圾回收统计
  - `/spark tps` 查看服务器 TPS/MSPT
  - 后台分析器（background profiler）自动持续采样
- **进度/奖励**：无直接进度关联。
- **摩擦**：无额外摩擦。后台分析器默认启用，对性能影响极小。
- **失败状态**：采样时间过长或堆转储过大可能导致临时卡顿或磁盘空间不足。

## 配置意图

`config/spark/config.json`：

```json
{
  "_header": "spark configuration file - https://spark.lucko.me/docs/Configuration",
  "backgroundProfiler": true
}
```

- **`backgroundProfiler: true`**：启用后台分析器，持续低频率采样 CPU 使用情况。这是唯一显式配置项，目的是在性能问题发生时保留历史采样数据，便于事后回溯根因。

`config/spark/tmp/` 与 `config/spark/tmp-client/` 为运行时临时目录，包含 `about.txt` 等自动生成文件。这些属于运行时噪音，不作为设计组件记录。

## 兼容性与性能

- **客户端/服务端范围**：BOTH。生产环境强烈建议在两侧都安装。
- **存档影响**：无直接存档影响。采样数据与堆转储文件可能占用磁盘空间，需定期清理。
- **已知不兼容**：当前批次中无已知冲突。
- **资源成本**：后台分析器开销极低；主动采样与堆转储会短暂增加 CPU/内存占用。
- **缓解措施**：
  - 后台分析器保持启用，用于长期趋势监控
  - 主动采样建议选择低峰期或短暂采样窗口
  - 定期清理 `config/spark/tmp*` 目录，避免临时文件堆积

## 风险与开放问题

1. **隐私与数据安全**：spark 生成的报告链接上传到 spark 官方服务，服务器运营者需了解此行为并决定是否允许。
2. **磁盘空间管理**：长期启用后台分析器会产生临时文件，需纳入日志/缓存清理规范。
3. **与性能优化模组的协同定位**：spark 帮助发现瓶颈，但优化需依赖 C2ME、Lithium、ModernFix 等模组。未来建立"spark 报告 → 优化决策 → 验证"的闭环流程。
