## 设计作用

Beautified Chat Server 是整合包的服务器端聊天格式化组件，为所有玩家消息统一添加时间戳、着色方案和可选的等级前缀。它是 ProjectNautic 聊天体验栈的"排版层"，确保无论玩家来自哪个平台、使用何种客户端，看到的聊天信息都具有一致的视觉结构和可读性。

该模组属于**实用型服务端组件**，支撑"服务器治理"与"信息可读性"设计方向：时间戳帮助玩家追踪事件顺序，颜色区分降低阅读疲劳，等级系统为未来社区管理预留扩展空间。

## 玩家可见行为

- **可发现性**：玩家进入服务器后，聊天框中的每条玩家消息都会带有 `HH:mm | 玩家名: 消息` 的统一格式，时间戳为灰色，玩家名为深绿色，消息为白色。
- **交互**：玩家无法直接通过客户端交互改变格式；服务器管理员可通过 `/bcs` 命令动态管理玩家颜色、等级（ranks）和自定义格式。
- **进度/奖励**：无直接进度关联；等级/称号系统未来可用于公会或贡献者标识。
- **摩擦/失败**：若服务端配置格式字符串错误，可能导致聊天消息解析异常或颜色代码泄露；等级数据未配置时，格式回退到无称号的基础样式。

## 集成关系

- **依赖**：NeoForge（required, SERVER）；运行时可能依赖 Collective 库（常见前置）。
- **冲突**：与同样重写聊天格式的服务端插件/模组冲突；客户端侧的聊天格式化模组若与服务端格式叠加，可能产生重复的时间戳或颜色代码。
- **协作**：与 No Chat Reports 共同决定聊天消息通道；与 Chat Heads 协作时需保证 `chatMessageFormat` 中 `%username%` 可被正确解析；与 ChatNotify 协作时，通知触发词会基于格式化后的消息文本进行匹配。
- **配方/标签**：无。
- **任务**：无。
- **世界生成**：无。
- **UI**：无新 GUI，仅修改聊天消息文本内容。
- **资源**：无特殊资源需求。
- **脚本**：无 KubeJS 交互；未来可通过服务器命令或数据包扩展等级系统。
- **加载顺序假设**：标准 SERVER 端模组；应在聊天相关 CLIENT 端渲染器之前完成消息格式化。

## 配置意图

### `config/beautifiedchatserver.json5`

- `showRankTitles: true`、`useRankColours: true`：启用等级称号与等级颜色，为后续社区管理（如 VIP、管理员、公会会长）预留可视化入口。
- `chatMessageFormat: "%timestamp% | %username%: %chatmessage%"`：统一聊天格式为"时间戳 | 玩家名: 消息"，结构清晰，便于 Chat Heads 等模组解析玩家名位置。
- `timestampFormat: "HH:mm"`：仅显示小时和分钟，避免时间戳过长挤压聊天内容；符合多数玩家的阅读习惯。
- `rankTitleFormat: "[%rank] "`：等级称号显示为 `[等级] 玩家名` 形式，与常见服务器惯例一致。
- `chatTimestampColour: 8`（dark_gray）：时间戳使用深灰色，视觉层级低于玩家名和消息内容。
- `chatUsernameColour: 2`（dark_green）：玩家名使用深绿色，在默认聊天背景下辨识度高且不刺眼。
- `chatMessageColour: 15`（white）：消息正文保持白色，最大化可读性。
- `chatOtherSymbolsColour: 7`（gray）：格式中的分隔符等符号使用灰色，避免喧宾夺主。

### `config/beautifiedchatserver/players.json`

- 当前为空对象 `{}`：尚未为任何玩家单独设置颜色或格式，未来可用于记录玩家自定义颜色。

### `config/beautifiedchatserver/ranks.json`

- 当前为空对象 `{}`：尚未定义任何等级（ranks），保留为社区运营阶段扩展使用。

## 兼容性与性能

- **客户端/服务端范围**：SERVER。纯服务端模组，客户端无需安装；所有格式由服务端计算后随聊天包下发。
- **存档影响**：无直接世界存档影响；玩家和等级数据存储在上述 JSON 配置文件中。
- **已知不兼容**：与其他服务端聊天格式化插件（如 EssentialsX Chat、LuckPerms 聊天后缀组件）可能重复格式化；No Chat Reports 的加密聊天若启用，可能与此模板的字符串替换逻辑冲突。
- **资源成本**：极低，仅在发送聊天消息时进行字符串拼接和颜色代码注入。
- **缓解措施**：保持 SERVER 端单一格式化权威，不在客户端安装额外聊天格式化模组；空白的 ranks/players 文件作为占位，未来扩展时避免格式冲突。

## 风险与开放问题

1. **格式字符串与 Chat Heads 的耦合**：当前格式将 `%username%` 放在固定位置，若未来调整为更复杂的布局（如称号在名称后），可能导致 Chat Heads 的头像匹配失败，需要在聊天组件中联动测试。
2. **等级系统尚未启用**：`ranks.json` 为空，社区管理相关的称号颜色体系还未建立，若服务器上线前需要分级展示，需补充配置并验证命令权限。
