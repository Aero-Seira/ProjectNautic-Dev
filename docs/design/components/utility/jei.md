# Just Enough Items (JEI)

## 记录

- 类型：mod
- 状态：active
- 证据可信度：verified
- 来源路径：`mods/[JEI物品管理器] jei-1.21.1-neoforge-19.38.0.366.jar`
- 标识与版本：mod_id = `jei`；版本 `19.38.0.366`
- 加载器或包格式：NeoForge 21.1.235

## 设计作用

JEI 是整合包**QoL（生活质量）核心**组件。作为物品与配方查看器，它为玩家提供了探索模组内容的唯一入口。

支持的设计支柱：
- **UI 一致性**：JEI 的配方查看界面是全整合包最频繁使用的交互界面，其布局、快捷键和搜索行为将直接影响玩家对整合包的第一印象。
- **可扩展性（KubeJS）**：所有 KubeJS 自定义配方、魔改配方、隐藏配方、条件配方都必须正确注册到 JEI 中暴露给玩家。如果配方仅在合成台可用但 JEI 不显示，等同于不存在。

## 玩家可见行为

### 可发现性
- 玩家打开任意物品栏（背包、箱子、合成台）时，JEI 物品列表自动显示在右侧。
- 鼠标悬停物品按 `R` 显示所有使用该物品的配方，按 `U` 显示所有产出该物品的配方。
- 点击合成台中的 `+` 按钮可将配方自动填充到合成网格（前提是背包中有足够材料）。

### 交互
- 底部搜索栏支持即时过滤，支持 `@modid` 按模组过滤、按物品名称过滤、按标签过滤（如 `#logs`）。
- 配方查看器支持翻页浏览多个配方变体，支持查看燃料燃烧时间、熔炉配方、酿造配方等。
- 创造模式物品栏可通过 JEI 快捷获取物品（若服务器开启 cheat mode）。

### 进度与奖励
- JEI 本身不涉及进度系统，但它是玩家理解所有进度所需材料的必要工具。

### 摩擦与失败状态
- 若 KubeJS 自定义配方未正确注册到 JEI，玩家将无法发现新配方，导致进度卡死。
- 若服务端与客户端 JEI 配置不一致（如 cheat mode 状态），可能导致客户端显示可用但实际服务端拒绝操作。
- 大量物品注册可能导致 JEI 初始化时间延长，低配机器上首次打开物品栏有明显卡顿。

## 集成关系

### 依赖
- **NeoForge** `>=21.0.118-beta`（required, BOTH）
- **Minecraft** `[1.21, 1.21.1)`（required, BOTH）
- **KubeJS**（间接关系）：KubeJS 自定义配方必须调用 JEI 集成 API 注册/隐藏配方。KubeJS 将 JEI 列为可选客户端依赖（`>=19.25.0.322`），意味着 KubeJS 的脚本有能力在 JEI 存在时注册额外信息。
- **JER / JEED / JER Integration**（扩展插件）：Just Enough Resources 新增世界生成、生物掉落、村民交易等配方分类；Just Enough Effects Descriptions 新增状态效果说明分类；JER Integration 为 Create、Thermal、Tinkers' Construct 等科技模组预留数据补丁。这些插件均通过 JEI 的配方查看界面展示。

### 冲突
- 与 REI、EMI 等同类物品查看器不兼容。整合包只应保留一个配方查看器。当前选择 JEI 作为官方推荐方案。

### 配方与标签
- JEI 本身不添加新配方，但会读取并显示所有已注册配方。KubeJS 通过 `event.remove()` 和 `event.add()` 修改的配方会直接影响 JEI 的显示。
- `config/jei/blacklist.json` 可隐藏特定物品，用于清理创造模式标签页或隐藏未使用的中间物品。

### 任务与世界生成
- 无直接关联。

### UI
- JEI 右侧物品列表覆盖在所有原版/模组物品栏 GUI 上。模组作者需要确保其自定义 GUI 不与 JEI 区域重叠，或通过 API 告知 JEI 排除区域。

### 资源
- JEI 支持通过资源包替换其内置纹理和主题。

### 加载顺序
- JEI 在 NeoForge 的 `BOTH` 侧加载。服务端配置（如 cheat mode）会影响所有客户端行为。

## 配置意图

以下配置选择反映了整合包的设计意图，**非全部配置项的逐条记录**。

### `config/jei/jei-client.ini`（主配置，330 行）
- **搜索栏位置与行为**：搜索栏置于底部中央，这是玩家最直觉的位置。启用即时搜索（无需回车），减少操作步骤。
- **物品列表布局**：右侧单列显示，高度自适应屏幕。低分辨率屏幕上自动减少每页显示数量，避免与 GUI 元素重叠。
- **配方查看器动画**：启用平滑过渡动画，提升 UI 一致性体验。
- **Cheat Mode 控制**：客户端默认禁用 cheat mode。实际由服务端 `config/jei-server.toml` 控制是否允许。避免单机与服务器行为不一致。

### `config/jei-server.toml`
- **Cheat Mode**：当前配置为 `false`。整合包定位为生存向体验，不允许直接获取物品。未来若开设创造测试服，可单独调整此文件。
- **Give 命令**：使用 `/give` 而非直接设置物品栏，确保与权限插件兼容。

### `config/jei/blacklist.json`
- 当前为空（19 B），预留用于后续 KubeJS 魔改中隐藏未使用的中间产物或调试物品。

### `config/jei/ingredient-list-mod-sort-order.ini`
- 当前显式列出 `Minecraft`、`Xaero's Minimap`、`Xaero's World Map` 的排序位置。其余模组按默认字母顺序排列。该配置确保原版与地图模组物品在列表中处于预期位置，后续若添加大量 KubeJS 自定义物品或内容模组，可继续扩展此排序。

### `config/jei/ingredient-list-type-sort-order.ini`
- 定义物品、流体、状态效果等类型的排序优先级。当前保留默认顺序（`ItemStack`、`FluidStack`、`MobEffectInstance`），后续若整合包强调流体或效果类内容，可调整优先级。

### `config/jei/recipe-category-sort-order.ini`
- 控制配方分类的显示顺序。当前顺序为原版配方分类（合成、铁砧、熔炼、酿造等）在前，JEI 信息分类居中，JER 分类（地牢、附魔、生物、植物、村民、世界生成）与 JEED 效果说明分类在后。该排序确保玩家先看到核心配方，再查看扩展信息。

### `config/jei/jei-colors.ini`
- 配置搜索高亮、选中状态的颜色主题。当前使用默认暗色主题，与整合包整体 UI 色调保持一致。

## 兼容性与性能

### 客户端/服务端范围
- 标记为 `BOTH`，但实际功能以客户端为主。服务端仅用于同步 cheat mode 权限和配方数据。
- **必须**在服务端安装，否则客户端连接时会因缺失配方同步而显示不完整。

### 存档影响
- 无直接存档数据。JEI 不写入世界存档。
- 玩家客户端的搜索历史和收藏夹存储在本地配置中，换设备不保留。

### 已知不兼容
- **REI / EMI**：不能同时安装。若未来需要 EMI 的功能（如配方树），需评估是否迁移。
- 某些添加自定义合成网格的模组（如 Tinkers' Construct）若未正确实现 JEI 插件，其配方将不可查看。

### 资源成本
- JEI 初始化时需要索引所有物品、方块、流体和配方。模组数量越多，初始化时间越长。
- 物品列表为即时渲染，高分辨率纹理包会显著增加 GPU 负担。
- 搜索使用客户端缓存，内存占用与注册物品数量成正比。

### 缓解措施
- 限制同时安装的大型内容模组数量（已在设计阶段控制）。
- 引导玩家分配至少 4GB 内存给客户端。
- 使用 `blacklist.json` 隐藏不必要物品，减少索引负担。

## 风险与开放问题

1. **KubeJS 配方同步风险**：若 KubeJS 脚本在服务端 reload 后未重新注册 JEI 配方，客户端可能需要重进服务器才能看到更新后的配方。需要在开发流程中建立 reload 后验证 JEI 的规范。
2. **大规模内容扩展时的性能瓶颈**：若未来添加超过 50 个内容模组，JEI 的初始化时间可能成为玩家抱怨点。考虑届时是否引入分页加载或延迟索引优化。
3. **配置漂移风险**：客户端可以修改本地 `jei-client.ini`，但服务端无法强制覆盖。重要配置（如 cheat mode）必须依赖服务端 `jei-server.toml`。
