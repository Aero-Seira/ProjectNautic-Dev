## 设计作用

Chat Heads 是聊天窗口的视觉增强模组，在每条玩家聊天消息旁显示发送者的皮肤头像。它将原本纯文本的聊天流转化为更易识别、更具社交感的对话界面，让玩家在公会、交易或团队协作中一眼区分发言人。

该模组属于**美学/实用型组件**，支撑 ProjectNautic"信息可读性"与"社交亲和力"的设计方向：整合包强调多人互动与社区氛围，头像能显著降低群聊中的认知负荷，同时与 Modern UI 的视觉风格保持一致的精致感。

## 玩家可见行为

- **可发现性**：玩家打开聊天窗口或收到新消息时，每条玩家消息左侧会自动显示发送者的皮肤头像。
- **交互**：无额外交互，头像为纯视觉提示；模组会通过玩家 UUID 或启发式规则匹配发送者皮肤。
- **进度/奖励**：无。
- **摩擦/失败**：若服务器修改了聊天格式导致发送者名称无法被正确解析，头像可能无法显示或显示默认 Steve/Alex；离线模式服务器可能因 UUID 缺失导致头像匹配失败。

## 集成关系

- **依赖**：NeoForge（required, CLIENT）。
- **冲突**：与大幅重写聊天渲染或隐藏玩家名称的模组可能存在显示冲突；无已知严重冲突。
- **协作**：与 Beautified Chat Server 的聊天格式化、No Chat Reports 的消息通道、ChatNotify 的通知系统共同工作；需确保 Beautified Chat Server 的 `chatMessageFormat` 不改变玩家名称在消息中的可识别位置。
- **配方/标签**：无。
- **任务**：无。
- **世界生成**：无。
- **UI**：仅修改聊天屏幕的消息行渲染，添加头像绘制。
- **资源**：依赖玩家皮肤的网络缓存（默认从 Mojang/服务器皮肤服务获取），首次加载时可能有轻微网络请求。
- **脚本**：无 KubeJS 交互。
- **加载顺序假设**：标准 CLIENT 端模组；聊天渲染应在 Beautified Chat Server 之后进行，以避免格式化标签干扰头像定位。

## 配置意图

### `config/chat_heads.json5`

- `renderPosition: "BEFORE_NAME"`：头像渲染在玩家名称之前，与原版的"头像-名称-内容"阅读顺序一致，最直观。
- `senderDetection: "UUID_AND_HEURISTIC"`：优先使用 UUID 精确匹配发送者，若服务器未提供 UUID 则回退到启发式名称匹配，覆盖大多数多人环境。
- `smartHeuristics: true`：启用智能启发式规则，减少名称相似或前缀/后缀干扰导致的误匹配。
- `handleSystemMessages: true`：允许模组尝试为系统消息附加头像（若可识别），保持聊天流视觉统一。
- `offsetNonPlayerText: true`：非玩家消息（如系统提示）整体右移，避免与玩家消息头像列不对齐造成视觉错乱。
- `threeDeeNess: 0.0`：关闭头像的 3D 倾斜效果，保持与 Modern UI 扁平、清晰的视觉风格一致。
- `drawShadow: true`：保留头像阴影，增加在明亮聊天背景下的可读性。
- `nameAliases: {}`：当前未配置玩家别名映射，未来若服务器使用昵称插件可在此维护。

## 兼容性与性能

- **客户端/服务端范围**：CLIENT。仅影响客户端聊天渲染，无需服务端安装。
- **存档影响**：无。
- **已知不兼容**：与完全替换聊天 GUI 的模组（如某些 RPG 聊天模组）可能冲突；极端自定义资源包若修改聊天行高度，可能导致头像错位。
- **资源成本**：
  - CPU：很低，仅聊天打开时渲染小尺寸头像。
  - 网络：首次见到某玩家时可能请求皮肤，正常游戏流量外可忽略。
  - 内存：缓存头像占用少量显存/内存。
- **缓解措施**：通过 `UUID_AND_HEURISTIC` 和 `smartHeuristics` 降低匹配失败率；关闭 3D 效果减少渲染开销。

## 风险与开放问题

1. **服务器格式化导致匹配失败**：若 Beautified Chat Server 后续调整格式导致玩家名称位置变化，Chat Heads 可能无法正确显示头像，需在聊天相关配置变更后回归验证。
2. **皮肤请求延迟**：在网络较差或皮肤服务不可达时，头像可能短暂显示默认皮肤，影响即时可识别性。
