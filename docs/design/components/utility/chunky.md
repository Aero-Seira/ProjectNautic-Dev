## 设计作用

Chunky 是 ProjectNautic 整合包的区块预生成管理工具，允许管理员通过命令行划定区域并提前生成区块，从而将世界生成开销从玩家实际探索时转移到服务器维护窗口。该模组直接支撑整合包**长线运营优先**与**性能优先**两大设计支柱：预生成可显著减少玩家探索新区域时的卡顿与 MSPT 峰值，是长期运营服务器的基础运维工具。

Chunky 标记为 `BOTH`，但主要用于服务端。客户端安装后可通过单人世界命令使用。

## 玩家可见行为

- **可发现性**：普通玩家通常不会直接与 Chunky 交互；管理员通过聊天框命令 `/chunky` 启动、暂停、取消预生成任务。
- **交互**：
  - `/chunky world <world>` 选择目标维度
  - `/chunky radius <radius>` 设置预生成半径
  - `/chunky start` 开始预生成
  - `/chunky pause`、`/chunky continue`、`/chunky cancel` 管理任务
- **进度/奖励**：无直接进度关联。
- **摩擦**：预生成过程会占用大量 CPU 与磁盘 I/O，管理员需在低峰期执行。
- **失败状态**：半径设置过大或服务器资源不足可能导致长时间任务、磁盘空间耗尽或 TPS 下降。

## 配置意图

`config/chunky/config.json`：

```json
{
  "version": 2,
  "language": "en",
  "continueOnRestart": false,
  "forceLoadExistingChunks": false,
  "silent": false,
  "updateInterval": 1
}
```

- **`language: "en"`**：命令与提示使用英文。这与整合包当前默认语言设置一致，便于管理员在文档中引用命令输出。
- **`continueOnRestart: false`**：服务器重启后不自动继续预生成任务。这是安全默认值，避免意外重启后自动占用资源；管理员需手动决定何时恢复任务。
- **`forceLoadExistingChunks: false`**：不强制加载已存在区块，避免对已有建筑或结构造成不必要的加载压力。
- **`silent: false`**：预生成进度会输出到控制台与聊天框，便于管理员监控。
- **`updateInterval: 1`**：每生成一个区块更新一次进度，提供较精细的进度反馈。

该配置体现"工具可用但需人工监管"的设计意图：Chunky 是强大的运维工具，默认不自动运行，避免对服务器造成意外负载。

## 兼容性与性能

- **客户端/服务端范围**：BOTH，但主要用于服务端。生产环境建议在服务端安装。
- **存档影响**：预生成会写入世界存档，增加存档体积。建议预生成前备份，并合理规划半径。
- **已知不兼容**：当前批次中无已知冲突。
- **资源成本**：预生成期间 CPU、内存与磁盘 I/O 占用极高，可能导致 TPS 下降。建议在低峰期或使用 `pause`/`continue` 分段执行。
- **缓解措施**：
  - 从小半径开始逐步扩展，避免一次性生成过大区域
  - 配合 C2ME 的并行世界生成优化可提高效率
  - 使用 `silent` 或提高 `updateInterval` 减少日志 spam

## 风险与开放问题

1. **存档体积膨胀**：预生成会显著增加世界存档大小，需提前规划磁盘容量。
2. **服务器负载管理**：预生成任务对服务器资源消耗大，需要制定运维规范（如何时运行、如何暂停）。
3. **与自定义维度/世界生成模组的兼容性**：若未来添加自定义维度或世界生成模组，需验证 Chunky 是否能正确在这些维度中预生成。
