## 设计作用

Cloth Config 是配置屏幕 API，为依赖它的模组提供统一的配置界面构建器、字段序列化与本地化支持。它在包内属于 UI 基础设施层：玩家通过依赖模组打开的“设置”界面，实际上由 Cloth Config 渲染与处理。

在 ProjectNautic 中，它支撑**UI 一致性**与**可访问性**支柱，使不同模组的配置界面在布局、滚动、搜索与保存行为上保持一致，减少玩家学习成本。

## 玩家可见行为

普通玩家不会直接看到 Cloth Config 的名称，但会在依赖它的模组（如聊天动画、效果视觉）的配置界面中体验到统一的列表、滑块、开关与分类风格。

- **配方/标签**：无直接参与。
- **任务**：无。
- **世界生成**：无。
- **UI**：提供统一的配置屏幕实现。
- **资源**：无。
- **脚本**：无 KubeJS 脚本交互需求。
- **加载顺序假设**：标准。Cloth Config 需在消费者之前完成加载。

## 配置意图

Cloth Config 本身不暴露 Minecraft 级别的配置文件。其行为由以下因素决定：

- **版本锁定**：effectual 要求 Cloth Config `>=15.0.140`。当前安装的 `15.0.140` 满足此约束。更新 API 前需确认所有消费者的兼容性声明。
- **界面风格**：包内未对 Cloth Config 的默认主题做额外覆盖，保持其原生视觉风格。

## 兼容性与性能

- **客户端/服务端范围**：BOTH。配置屏幕在客户端渲染，但配置序列化与同步逻辑在两侧都可能使用。
- **存档影响**：无直接存档影响。
- **已知不兼容**：当前批次中无已知冲突。
- **资源成本**：库模组开销极低，配置屏幕仅在打开时占用少量 CPU/GPU 资源。
- **缓解措施**：保持与依赖它的模组版本同步；避免手动删除或替换。

1. **版本绑定**：Cloth Config 的更新与 chatimpressiveanimation、effectual 等消费者紧密耦合。
2. **风格碎片化风险**：若未来同时引入 YACL、MidnightLib 等其他配置库，可能出现多套配置界面风格并存的情况，需要在设计层面统一。
